# 补包系统下一步建议（服务端 + 客户端）

## P0（先做，影响稳定性）

1. **把运行参数外置**（IP、端口、缓冲区大小、超时、重试次数）
   - 当前参数硬编码在 `main.cpp` / 头文件中，切换环境和多实例部署会很痛苦。
   - 建议支持 `yaml/json + 命令行覆盖`。

2. **完善协议的字节序与版本机制**
   - `RetransHeader/RetransRequestBody` 目前直接按内存布局收发，默认依赖同端 CPU 字节序。
   - 建议：
     - 所有整数字段采用网络字节序；
     - 增加 `version` 字段；
     - 增加 `request_id`，便于链路追踪。

3. **收发路径加可观测性**
   - 至少输出并上报：
     - 主流接收率、重传请求数、重传命中率、最终恢复率、超时放弃率；
     - 各线程队列长度和环形缓冲区水位。
   - 建议 Prometheus 指标化。

4. **系统化压测和故障注入**
   - 构建丢包/乱序/突发丢包场景，明确在不同丢包率下的恢复曲线。
   - 自动回归：每次改动后跑 5~10 分钟并输出恢复 KPI。

## P1（第二阶段，提升可维护性）

1. **请求聚合策略**
   - 当前客户端是按单包请求补包。
   - 建议合并连续缺包区间，按窗口请求，减少 PPS 和系统调用。

2. **发送节奏控制**
   - `TsOutputSender` 目前是基础节奏，建议按目标码率动态整形，避免瞬时突发。

3. **优雅退出与线程生命周期管理**
   - 统一 stop 信号、socket 关闭顺序、线程退出等待，减少僵死风险。

4. **日志分级和限频**
   - 高频路径（每包日志）应做降采样或限频，否则高码率会压垮 CPU 与磁盘。

## P2（产品化）

1. **多客户端支持策略**
   - 明确是单播回包还是组播回包、是否需要会话管理和鉴权。

2. **安全能力**
   - 至少考虑请求源校验、限速、防重放。

3. **部署与运维资产**
   - systemd 单元、健康检查、启动脚本、配置模板、运行手册。

## 建议的近期里程碑（两周）

- **第 1 周**：参数外置 + 协议字节序规范 + 基础指标打点。
- **第 2 周**：请求聚合 + 压测脚本 + 故障注入回归。

完成后再评估更复杂的 FEC/多路径方案，避免过早复杂化。
